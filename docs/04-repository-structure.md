## 2. Layer Conventions

### Data Layer (`lib/data/`)

| Rule | Rationale |
|---|---|
| DAOs return Drift-generated data classes only | Keep Drift types contained |
| Mappers are pure static functions or extension methods | No side effects, easy to test |
| Repository impls catch all exceptions and return `Either<Failure, T>` | Uniform error handling boundary |
| One DAO per aggregate root | Prevents God-DAO anti-pattern |
| No business logic in repositories | Repositories are data plumbing, not decision-makers |

### Domain Layer (`lib/domain/`)

| Rule | Rationale |
|---|---|
| Zero Flutter imports | Framework independence |
| Entities are immutable (`freezed`) | Predictable state, safe for provider/notifier state updates |
| Repository interfaces are abstract classes | Inversion of control |
| Use cases have a single `call()` method | Single Responsibility Principle |
| Use cases return `Future<Either<Failure, T>>` | Consistent async error handling |
| One class per file | Navigation and discoverability |

### Presentation Layer (`lib/presentation/`)

| Rule | Rationale |
|---|---|
| Providers depend on use cases, never repositories | Enforces domain boundary |
| One Notifier per feature concern (list, detail, form) | Keeps provider scope manageable |
| Pages are thin вЂ” delegate to notifier for logic, widgets for rendering | Separation of concerns |
| Widgets under `widget/` are reusable within the feature | Co-location over centralization |
| Shared widgets go in `common/widget/` | Cross-feature reuse |
| Providers use code generation to ensure type safety and lifecycle management | Auto-dispose and type safety |

### Core (`lib/core/`)

| Rule | Rationale |
|---|---|
| No feature-specific code | Core is shared infrastructure |
| `Failure` hierarchy defined here | Used by all layers |
| Extensions organized by type they extend | Easy discovery |
